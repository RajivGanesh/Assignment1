4 Size and Performance

This part of the document references the sizing and timing requirements in the FLOP v1 cloud application. The system can support up to 4000 users simultaneously with the current workspace package it is implemented with on Oracle’s servers. Oracle offers unlimited bandwidth with a throttle at a certain cap, which means that the average interaction latency is very low. This means that the system should not take more than 30 seconds to complete any task that users at any level of the hierarchy may assign.

Disk space is shared among the clients, because they are all operating on a single data source. The database tables are compressed daily, giving an average table size of 2 kilobytes per 20 records without blob elements. System RAM is also shared among the users, but Oracle servers offer virtualization of unused disk space that augment gaps in cache demand. Having the database normalized to 3NF results in better performance, and a better data storage economy.

In the database currently, there is an automated sequence that generates user ID’s and keeps track of them so that it is easier know the database’s data storage limits. Currently the sequence caps the number of users that can be registered at one million accounts. In the pool of the one million users, the sequence also constrains half a million of these users to be commercial users, and the other half to be agricultural users. This can be re-balanced at anytime by editing the script. All the tables in the database are capped to accept a million records, with the exception of the OrderSell, and OrderBuy tables which have their capacity increased to two million records.

As stated before in the system requirements portion of the document, the system downtime is very miniscule, as it can never be down more than once in a 24 hour period, and it has scheduled maintenance on Friday nights outside of peak time. The application is live 24/7 and can be accessed from anywhere, with the only limit being the user’s browser, and bandwidth.
